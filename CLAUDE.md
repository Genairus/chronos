# Chronos Toolchain — Project Instructions for Claude Code

## What Is Chronos

Chronos is a domain-specific language (DSL) for specifying software requirements as structured, compilable models. The compiler validates `.chronos` files against 50+ rules and generates artifacts (Markdown PRDs, Jira backlogs, TypeScript types, Mermaid state diagrams, test scaffolds).

## Language Reference

See `docs/quick-reference.md` for the complete language syntax and all 15 shape types.
See `docs/ai-bot-requirements-howto.md` for a comprehensive language specification with examples and diagnostic code reference.
Grammar source: `chronos-parser/src/main/antlr/com/genairus/chronos/parser/Chronos.g4`.

## Module Architecture

8 modules with strict dependency boundaries enforced by `build.gradle.kts`:

```
chronos-core          (zero deps — Span, Diagnostic, QualifiedName, NamespaceId, ShapeId)
├── chronos-ir        (api dep on core — IrModel, IrShape sealed hierarchy, TypeRef, SymbolRef)
├── chronos-parser    (core — ANTLR grammar, LoweringVisitor → SyntaxModel/SyntaxDecl DTOs)
chronos-validator     (ir, core — ChronosValidator with 50+ CHR-xxx checks)
chronos-compiler      (core, ir, parser, validator — 7-phase pipeline, multi-file compilation)
chronos-generators    (core, ir — MarkdownPrdGenerator, JiraCsvGenerator, etc.)
chronos-artifacts     (core, ir, compiler — IrArtifactEmitter, IrModelSerializer)
chronos-cli           (all above — PicoCLI commands: prd, validate, build, generate, select, diff)
```

Boundary rules are in `build.gradle.kts` `allowedProjectDeps` map. `./gradlew check` enforces them.

## Build & Test

```bash
./gradlew build                    # compile + test all modules
./gradlew :chronos-cli:test        # test CLI module only
./gradlew :chronos-validator:test  # test validator only
./gradlew check                    # full check including architecture boundary verification
```

Java 21 required. Version: `0.1.0` (set in root `build.gradle.kts` line 122).

## CLI Patterns

- Framework: PicoCLI 4.7.5
- Commands implement `Callable<Integer>` for exit codes
- Parent command: `ChronosCli` with `@ParentCommand ChronosCli parent` + `@Spec CommandSpec spec`
- Console output: `parent.console(spec.commandLine().getOut(), spec.commandLine().getErr())`
- Tests: `new CommandLine(new ChronosCli()).setOut(...).setErr(...).execute(args)`
- Version: `mixinStandardHelpOptions = true` provides `-V` (uppercase, to avoid conflict with `validate -v`)
- `ChronosVersion.java` is generated by Gradle task `generateVersionSource`

## Sealed Switch Exhaustiveness

When adding a new `IrShape` subtype, ALL exhaustive switches must be updated:
- `IrRefWalker` (chronos-ir)
- `IrModelSerializer` (chronos-ir)
- `ModelProjection` (chronos-cli)
- `SelectCommand` (chronos-cli)
- `DiffCommand` (chronos-cli)

Grep for `return switch (shape)` to find them all.

## Compiler Pipeline (7 Phases)

1. ParseAndLower — ANTLR parse → SyntaxModel
2. CollectSymbols — build symbol table
3. BuildIrSkeleton — SyntaxModel → IrModel
4. TypeResolution — resolve NamedTypeRef (transformer, returns new model)
5. CrossLinkResolution — resolve SymbolRefs (inheritance, relationships)
6. Validation — ChronosValidator (CHR-001 through CHR-053)
7. FinalizeIr — check for any remaining unresolved refs (CHR-012)

Multi-file: `compileAll()` does IndexPass → sort by path → per-file Phases 3-6 → unit-level finalize.

## Parser API

- New: `new ChronosParserFacade().parse(source, filename)` → `SyntaxModel` (instance method, not static)
- Legacy: `ChronosModelParser.parseFile(Path)` → throws checked IOException + ChronosParseException
- `SyntaxDecl` sealed interface with 13 permits (all in `com.genairus.chronos.syntax`)
- `LoweringVisitor` in `com.genairus.chronos.parser.lowering`

## Diagnostic System

- `Diagnostic` record in `com.genairus.chronos.core.diagnostics`: `.code()`, `.severity()`, `.message()`, `.span()`, `.pathOrNull()`
- `DiagnosticCollector`: `.add()`, `.error()`, `.warn()`, `.info()`, `.all()`, `.hasErrors()`
- Severity: `DiagnosticSeverity.ERROR`, `WARNING`, `INFO`
- Codes CHR-001 through CHR-053 + CHR-W001 (see `DiagnosticCodeRegistryTest` for authoritative list)

### Diagnostic Code Ownership

- CHR-012 = exclusively FinalizeIrPhase (unresolved SymbolRef) — validator must never emit it
- CHR-016 = exclusively ImportResolver (unknown import target) — validator must never emit it
- CHR-005 = both ChronosValidator (per-file) and GlobalSymbolTable (cross-file)
- CHR-048/049 = ChronosValidator (composition single-owner / field override)

## IR Types

- `IrShape` sealed interface: EntityDef, ShapeDef, EnumDef, ListDef, MapDef, ActorDef, PolicyDef, JourneyDef, RelationshipDef, InvariantDef, DenyDef, ErrorDef, StateMachineDef, RoleDef, EventDef
- `TypeRef` sealed: PrimitiveType, NamedTypeRef, ListTypeRef, MapTypeRef
- `SymbolRef`: resolved (has ShapeId) or unresolved (has QualifiedName + SymbolKind)
- `IrModel`: record with namespace, shapes list, imports list, finalized flag

## Key Conventions

- Package root: `com.genairus.chronos`
- `chronos-model` has been REMOVED — all IR types live in `chronos-ir`
- `chronos-ir` uses `api(project(":chronos-core"))` so consumers get core transitively
- CLI needs explicit `implementation(project(":chronos-core"))` for direct core imports
- Never use `.contains("ir")` for package checks — `"genairus".contains("ir") == true`
- `Span.UNKNOWN.sourceName()` → `"<unknown>"`, `.startLine()` → `0`

## Language Gotchas (for writing .chronos test fixtures)

- `description` is a contextual keyword — cannot be used as field name in entity/shape
- Entity invariant expressions reference only direct fields (no dot-path navigation)
- Global invariant scope must reference entities in the same namespace
- `outcomes` block is required on journeys (CHR-002)
- Invariant severity: error/warning/info. Deny/error severity: critical/high/medium/low.
- Enum values are optional integers only: `MEMBER = 1` (not strings)
